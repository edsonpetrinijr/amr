### 2026-06-09 ~06:45 — M4 Boot destravado: "M4 Running Version" instalada (trick-m4-6.13.4 + JDK 17)
- **Done:** No notebook corporativo (Caterpillar), o **M4 Boot 2.1** (launcher SEER, abre em `localhost:6800`) não mostrava nenhuma **"M4 Running Version"** e o botão de baixar versão dava **erro 500**. Investigação de causa raiz no log do backend: `获取线上 M4 版本列表失败: fetch failed → ECONNRESET`. O backend Node usa **`fetch` (undici)**, que **ignora as variáveis de proxy** e tenta conexão direta — bloqueada pelo proxy corporativo (`ch3proxy.cat.com:9090`, PAC `pac.cat.com`). Logo, **todos os downloads online do wizard estão quebrados nessa rede** (lista de versões, versão e JDK). Solução **sem admin**: baixei manualmente via PowerShell (que passa pelo proxy) a versão mais recente **trick-m4-6.13.4** (266 MB) + **JDK 17.0.14 LTS** (177 MB), extraí em `C:\Users\junioeg\m4\{trick-m4-6.13.4, jdk}` e apontei o `m4home` do serviço (que roda como **LocalSystem**) via `POST /api/storage/set/m4home`. Confirmado pela própria API: `release/local` → **6.13.4** detectada; `java` → **bundle JDK 17** reconhecido.
- **Decided:** Nessa rede corporativa, **não usar os downloads online do M4 Boot** (undici não respeita proxy) — **provisionar versão + JDK manualmente** em `~\m4\` é o caminho confiável. Antes disso, dois tropeços resolvidos: rodar o `.exe` em `C:\Program Files` dava **EPERM** (release na pasta protegida) → mover p/ pasta do usuário; e **EADDRINUSE 6800** porque o app **já roda como serviço Windows via nssm** ("M4 Boot Service", auto-start) — não precisa abrir o `.exe` na mão, usa-se o `localhost:6800` do serviço.

JDK 17 já reconhecido. **Bloqueio conhecido:** o 1º uso do M4 pede **autorização/licença** (machine code → arquivo `m4-license`, contato SEER/Xiangong). Melhoria futura (fora do nosso controle): app suportar proxy no undici.
